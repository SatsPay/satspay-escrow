# satspay-escrow

The core smart contract of the SatsPay protocol. It holds sBTC on behalf of recipients who haven't yet claimed their funds, enforces expiry and refund logic, and releases funds only to verified claimants.

Deployed on: **Stacks Testnet** (hackathon) → **Stacks Mainnet** (v1.0)

---

## Overview

When a sender wants to pay someone who doesn't have a Stacks wallet yet (or whose wallet address is unknown), they send to a **phone number hash** instead of a wallet address. The escrow contract holds the sBTC safely until the recipient claims it — or returns it to the sender after 30 days.

```
Sender wallet
    │
    │  send-to-phone(phone-hash, amount, expiry)
    ▼
satspay-escrow ◄──── holds sBTC ────► Recipient claims → recipient wallet
    │
    └── after expiry → reclaim() → sender wallet
```

## Contract Architecture

### Storage (Data Maps & Vars)

```clarity
;; Stores each pending transfer
(define-map pending-transfers
  { claim-id: (buff 32) }
  {
    sender: principal,
    phone-hash: (buff 32),
    amount: uint,
    expiry-block: uint,
    claimed: bool
  }
)

;; Total sBTC held in escrow (for balance verification)
(define-data-var total-escrowed uint u0)
```

### Public Functions

#### `send-to-phone`
Called by the sender. Locks sBTC in the contract.

```clarity
(define-public (send-to-phone
  (phone-hash (buff 32))
  (amount uint)
  (claim-id (buff 32))
  (expiry-blocks uint))
```

**Parameters:**
| Parameter | Type | Description |
|---|---|---|
| `phone-hash` | `(buff 32)` | SHA-256 hash of the recipient's phone number (e.g. `+2348012345678`) |
| `amount` | `uint` | Amount of sBTC in micro-units (1 sBTC = 100,000,000 micro-sBTC) |
| `claim-id` | `(buff 32)` | Unique identifier generated by the backend, used to look up this transfer |
| `expiry-blocks` | `uint` | Number of Stacks blocks until the transfer expires (~10 min/block, so 4320 = 30 days) |

**What it does:**
1. Validates amount is greater than zero
2. Validates expiry is at least 144 blocks (~24 hours) in the future
3. Calls `satspay-sbtc-interface::transfer` to move sBTC from sender to this contract
4. Stores the transfer record in `pending-transfers`
5. Updates `total-escrowed`
6. Emits a `transfer-initiated` print event (picked up by backend webhook)

**Errors:**
| Code | Name | Meaning |
|---|---|---|
| `u100` | `err-invalid-amount` | Amount must be greater than zero |
| `u101` | `err-expiry-too-soon` | Expiry must be at least 144 blocks away |
| `u102` | `err-claim-id-taken` | A transfer with this claim-id already exists |
| `u103` | `err-transfer-failed` | sBTC transfer from sender failed (insufficient balance?) |

---

#### `claim`
Called by the recipient (or by the backend on their behalf after verifying the claim token). Releases sBTC to the recipient's wallet.

```clarity
(define-public (claim
  (claim-id (buff 32))
  (recipient principal))
```

**Parameters:**
| Parameter | Type | Description |
|---|---|---|
| `claim-id` | `(buff 32)` | The claim ID from the original `send-to-phone` call |
| `recipient` | `principal` | The Stacks address of the recipient's wallet |

**What it does:**
1. Looks up the transfer by `claim-id`
2. Verifies it exists and has not been claimed yet
3. Verifies it has not expired
4. Calls `satspay-sbtc-interface::transfer` to move sBTC from contract to recipient
5. Marks transfer as `claimed: true`
6. Updates `total-escrowed`
7. Emits a `transfer-claimed` print event

**Errors:**
| Code | Name | Meaning |
|---|---|---|
| `u200` | `err-claim-not-found` | No transfer with this claim-id exists |
| `u201` | `err-already-claimed` | Transfer has already been claimed |
| `u202` | `err-transfer-expired` | Transfer has passed its expiry block |
| `u203` | `err-claim-transfer-failed` | sBTC transfer to recipient failed |

---

#### `reclaim`
Called by the original sender after the transfer has expired. Returns sBTC to the sender.

```clarity
(define-public (reclaim (claim-id (buff 32)))
```

**Parameters:**
| Parameter | Type | Description |
|---|---|---|
| `claim-id` | `(buff 32)` | The claim ID from the original `send-to-phone` call |

**What it does:**
1. Looks up the transfer by `claim-id`
2. Verifies the caller is the original sender (`tx-sender == transfer.sender`)
3. Verifies the transfer has expired (current block > expiry-block)
4. Verifies it has not been claimed
5. Returns sBTC to the sender
6. Marks transfer as `claimed: true` (prevents double-reclaim)
7. Emits a `transfer-reclaimed` print event

**Errors:**
| Code | Name | Meaning |
|---|---|---|
| `u300` | `err-claim-not-found` | No transfer with this claim-id exists |
| `u301` | `err-not-sender` | Caller is not the original sender |
| `u302` | `err-not-expired` | Transfer hasn't expired yet — recipient can still claim |
| `u303` | `err-already-claimed` | Transfer already claimed or reclaimed |

---

### Read-Only Functions

#### `get-transfer`
Returns full details of a pending transfer.

```clarity
(define-read-only (get-transfer (claim-id (buff 32)))
  (map-get? pending-transfers { claim-id: claim-id }))
```

#### `is-claimable`
Returns `true` if a transfer exists, hasn't been claimed, and hasn't expired.

```clarity
(define-read-only (is-claimable (claim-id (buff 32)))
```

#### `get-total-escrowed`
Returns total sBTC currently held by the contract.

```clarity
(define-read-only (get-total-escrowed) (var-get total-escrowed))
```

## Events (Print Statements)

The backend listens for these onchain events via Hiro API webhooks:

| Event | Emitted by | Payload |
|---|---|---|
| `transfer-initiated` | `send-to-phone` | `{ claim-id, phone-hash, amount, sender, expiry-block }` |
| `transfer-claimed` | `claim` | `{ claim-id, recipient, amount }` |
| `transfer-reclaimed` | `reclaim` | `{ claim-id, sender, amount }` |

## Security Considerations

### Phone Number Privacy
Phone numbers are **never stored onchain in plaintext**. The sender hashes the phone number client-side using SHA-256 before calling the contract:
```js
const phoneHash = sha256(normalizePhone("+2348012345678"))
// → stored as (buff 32) onchain
```
This means the blockchain reveals nothing about who is sending to whom.

### No Admin Key
There is no contract owner, no admin function, and no upgrade mechanism. Once deployed, the contract logic is immutable. Funds can only move via `claim` (to recipient) or `reclaim` (back to sender after expiry). No third party can touch the funds.

### Reentrancy
Clarity is not susceptible to reentrancy attacks by design — all state changes happen before external calls, and Clarity's execution model prevents recursive calls. However, the contract still marks transfers as claimed before executing the sBTC transfer, as a belt-and-suspenders measure.

### Expiry Logic
Expiry is measured in **Stacks block height**, not wall-clock time. Stacks produces approximately one block every 10 minutes. The minimum expiry is 144 blocks (~24 hours) and the recommended default is 4320 blocks (~30 days).

## Integration (Backend)

### Calling `send-to-phone` from Node.js

```typescript
import { makeContractCall, bufferCVFromString, uintCV } from '@stacks/transactions';
import { StacksTestnet } from '@stacks/network';
import { sha256 } from 'js-sha256';

const network = new StacksTestnet();

const phoneHash = Buffer.from(sha256('+2348012345678'), 'hex');
const claimId = Buffer.from(crypto.randomUUID().replace(/-/g, ''), 'hex');

const txOptions = {
  contractAddress: 'ST...',
  contractName: 'satspay-escrow',
  functionName: 'send-to-phone',
  functionArgs: [
    bufferCV(phoneHash),          // phone-hash
    uintCV(100000),               // amount (0.001 sBTC)
    bufferCV(claimId),            // claim-id
    uintCV(4320),                 // expiry (~30 days)
  ],
  senderKey: process.env.SENDER_PRIVATE_KEY,
  network,
};

const transaction = await makeContractCall(txOptions);
const result = await broadcastTransaction(transaction, network);
```

### Calling `claim` from Node.js

```typescript
const txOptions = {
  contractAddress: 'ST...',
  contractName: 'satspay-escrow',
  functionName: 'claim',
  functionArgs: [
    bufferCV(claimId),
    standardPrincipalCV(recipientAddress),
  ],
  senderKey: process.env.BACKEND_SIGNER_KEY,
  network,
};
```
